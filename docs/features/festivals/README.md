# Festivals (серии и выпуски)

Документация описывает **серии фестивалей** (бренд/линейка) и **выпуски** (конкретная итерация), а также их связь с событиями, Smart Update и Universal Festival Parser.

## Основные понятия

- **Серия (series)** — устойчивое название фестиваля без года/номера. Пример: «Кантата», «Открытое море», «Гаражка».
- **Выпуск (edition)** — конкретная итерация серии. Пример: «Кантата 2026», «Кантата — весна 2026», «XIV Кантата».
- **Псевдо‑фестивали** — праздники из `docs/reference/holidays.md`, которые также версионируются по годам.

В UI используем **короткое название серии**, а для навигации между выпусками показываем год/номер/сезон (edition‑label).

## Базовая модель данных

- Текущая таблица `festival` хранит **выпуски**.
- У каждого выпуска есть короткое имя серии (`name`) и подробное название (`full_name`).
- Дополнительная мета серий/выпусков хранится в полях серии или вычисляется из данных.

## Связь с событиями

- **Единственный инструмент создания/обновления событий — Smart Update.**
- `event.festival` всегда указывает на **выпуск** фестиваля.
- На страницах событий, месяца и выходных отображается ссылка на страницу фестиваля (уже реализовано).

## Детекция фестиваля в Smart Update

Детекция является частью **основного LLM‑разбора** (Gemma), отдельного запроса нет.

Результат должен содержать:
- `festival_context` = `festival_post` | `event_with_festival` | `none`.
- `festival` (короткое имя серии) и при наличии `festival_full` (полное название выпуска).
- ссылки/сигналы, полезные для дедупликации (сайт, соцсети, программа).

Поведение:
- **festival_post** → событие **не создаём**, источник попадает в фестивальную очередь.
- **event_with_festival** → событие создаём/обновляем через Smart Update, выставляем `event.festival`.

Если `event_with_festival` и выпуск не найден, создаётся **stub‑выпуск** (минимальные поля: серия + даты из события). При поступлении новых источников выпуск обогащается ссылками и описанием.

### Кейс‑матрица: пост о фестивале vs пост о событии

- Описание фестиваля/праздника целиком + программа/расписание/многодневность → `festival_post` (в очередь, без создания события).
- Анонс конкретного мероприятия с фразой «в рамках фестиваля/праздника» → `event_with_festival` (событие создаётся, `event.festival` выставляется).
- Анонс одного конкретного мероприятия внутри цикла/программы/фестивального источника (с собственной датой, временем, площадкой/маршрутом и билетом/регистрацией) не должен уходить в `festival_post`: это `event_with_festival`, если имя серии известно, иначе обычное событие. Списки материалов/условий участия внутри такого поста сами по себе не превращают его в multi-event программу.
- Telegram-посты фестиваля, где перечислено несколько конкретных dated program items, должны оставаться LLM-first: основной extractor получает шанс вернуть отдельные event rows по каждому пункту. Упрощённый schedule-rescue нельзя использовать как предварительный terminal `[]` для таких постов только из-за `schedule_like`, если есть явные festival/registration/lecture/library/museum signals.
- «День <…>» без программы и без признаков многодневности → обычное событие (`festival_context=none`).
- «День <…>» с короткой программой в один день → всё равно `festival_post` и program‑only (в очередь, без создания событий).

### Дополнительные триггеры: «День <…>» и «программа/расписание»

**«День <…>»** (например «День рыбака», «День защиты морских млекопитающих») — потенциальный фестивальный сигнал, если есть признаки программы, многодневности или нескольких площадок.  
Это **не означает**, что любой «день» автоматически фестиваль: «День открытых дверей», «День рождения», «День скидок» обычно остаются обычными событиями, если нет явной программы.

Правило матч‑серии:
- если «День <…>» совпадает с записью из `docs/reference/holidays.md`, используем этот псевдо‑фестиваль (выпуск по году);
- если совпадения нет, создаём серию с коротким названием «День <…>» и выпуск по датам текущей программы.
Для «День <…>» по умолчанию используем **выпуск с годом**, извлечённым из дат, даже если год не написан в названии, и не склеиваем с прошлогодними версиями.

**«Программа/расписание/план мероприятий/афиша»** — усилитель для `festival_post`, если одновременно выполняется одно из условий:
- указан диапазон дат (несколько дней);
- есть список активностей с несколькими временем/датами;
- присутствует явный маркер «программа фестиваля/праздника/мероприятий».

«Программа концерта» или «программа экскурсии» **не** является самостоятельным фестивальным сигналом без дополнительных признаков.  
Сигналы можно извлекать не только из текста, но и из OCR афиш/вложений.

## Дедупликация серии и выпусков (умная)

Используется комбинация детерминированных правил и LLM‑сигналов:
- нормализация имени (casefold, чистка кавычек, убираем «фестиваль/международный/городской» и т.п.);
- алиасы из `festival.aliases` и `festival_alias_pairs` (см. `docs/llm/prompts.md`);
- совпадение по официальным ссылкам/соцсетям;
- LLM‑матчинг на ближайшее каноническое имя.

Если серия найдена, выбирается **выпуск**:
- по явному году/номеру/сезону в `festival_full`;
- по кластеризации дат (см. следующий раздел), без фиксированного порога разрыва.

## Разделение выпусков без фиксированного порога

Если явного года/номера нет, выпуск определяется по **плотности дат** внутри серии:
- сортируем даты событий серии;
- ищем аномальные разрывы **внутри той же серии** (outlier‑gaps);
- границы выпусков ставятся на статистически выделяющихся разрывах.

Это позволяет корректно обрабатывать:
- фестивали на 1–2 дня с паузой в месяц;
- регулярные фестивали несколько раз в год;
- ежегодные фестивали, где выпуск длится через границу года.

## Псевдо‑фестивали (holidays)

- Праздники из `docs/reference/holidays.md` считаются сериями.
- Каждый год создаётся отдельный выпуск (склейки с прошлогодними быть не должно).
- Псевдо‑фестиваль используется **только если нет явного фестивального имени**.

## Фестивальная очередь

Очередь создаётся при `festival_post` и служит для комплексного разбора.  
Также Smart Update добавляет источники в очередь **best-effort** при `event_with_festival` (чтобы собрать/обновить страницу фестиваля), если удалось распознать имя фестиваля.

- VK‑посты анализируются локально (без Kaggle).
- Telegram‑посты обрабатываются **только через Kaggle**.
- Внешние сайты фестиваля — через Universal Festival Parser (Playwright + Gemma).
- Если TG/Kaggle прогон завершился ошибкой, но в `festival_queue` уже есть валидные `festival_name` + `source_text`, очередь делает fallback: создаёт/обновляет stub фестиваля из данных очереди и продолжает синхронизацию страницы фестиваля/индекса (в `result_json` фиксируется `monitor_error`).
- TG/VK соцсети фестиваля не выводим из URL поста-источника «по умолчанию»; сохраняем только явно подтверждённые ссылки (payload/UDS/сайт/явный self-link).
- Для Telegram: если `festival.tg_url` совпадает с каналом `source_post_url`, ссылка показывается только при явном подтверждении канала в БД (`telegram_source.festival_source=1` и серия совпадает с фестивалем).
- Для кейсов `День <...>` приоритет у явного названия из исходного текста/цитаты поста; если имя серии из parser/queue не заземлено в тексте поста, оно заменяется на явно указанное в источнике (чтобы не склеивать разные «Дни ...» между собой).

Примечание по Telegram Monitoring: если extractor пометил объект как `festival: true` (или `event_type: festival`) без строкового имени фестиваля, для постановки в очередь используется `title` (как имя серии).

Smart Update сообщает оператору:
- что источник поставлен в очередь;
- команду ручного запуска;
- статус автозапуска (вкл/выкл).

## Definition of Done (DoD) фестивальной очереди

Считаем задачу «фестивальная очередь отработала корректно», если для **каждого** выявленного фестиваля после обработки `/fest_queue` выполнено:

1) **Страница фестиваля в Telegraph**
   - есть обложка + галерея иллюстраций (минимум 2 изображения, если доступно);
   - обложка/галерея на странице фестиваля должны использовать preview‑friendly URL (Catbox не должен ломать Telegram cached preview страницы фестиваля);
   - если в snapshot остались только legacy Catbox‑картинки, страница должна сохранять безопасную обложку (для preview/cached_page) и не терять галерею целиком;
   - есть **короткое описание в 1 абзац** (LLM‑генерация допустима как fallback, если нет UDS/описания);
   - есть список «Мероприятия фестиваля» (или явная заглушка, если событий пока нет);
   - есть явная строка с количеством источников, использованных для сборки страницы фестиваля (`📚 Источников: N`, где `N >= 1` для нормального кейса);
   - в списке мероприятий у каждого события есть **короткое описание‑дайджест** (1 предложение; цель ≤ 12 слов, редкие исключения ≤ 16), используемое также в `/daily` и на страницах месяца/выходных;
   - short‑дайджест события должен приходить из LLM как 1 законченное предложение на 12–16 слов; рендер не должен показывать обрывы вида `...`/`…`;
   - есть ссылки минимум на **сайт** и/или **соцсети** (VK/TG), если они явно подтверждены (payload/UDS/сайт/явный self-link фестиваля);
   - `пост-источник` (VK/TG post URL) не публикуется на публичной странице фестиваля;
   - внешняя ссылка-источник допускается для канонического `source_url` фестиваля (сайт/страница программы), если он есть.
   - служебные маркеры навигации «ближайшие фестивали» **не должны быть видимы** на странице (используем только невидимые устойчивые маркеры для идемпотентного обновления).
   - fallback-рендер не должен терять кликабельность сервисных ссылок события (`подробнее`, `добавить в календарь`).

2) **Общая страница «Фестивали» (индекс)**
   - фестиваль присутствует на общей странице;
   - карточка/ссылка ведёт на корректную страницу фестиваля;
   - для фестивалей с обложкой она используется в карточке индекса.
   - `og:image`/превью Telegram не должны ломаться из‑за недоступного хостинга картинок (если нет «preview‑friendly» обложки, индекс должен использовать безопасную обложку).
   - в индексе для превью используются только «preview‑friendly» картинки (Catbox не используется как `og:image` и не должен становиться первой картинкой страницы).
   - индекс показывает **актуальные** фестивали (текущие и ближайшие в окне `FESTIVALS_UPCOMING_HORIZON_DAYS`, по умолчанию `120` дней), без «дальнего хвоста» старых/сезонно-далёких выпусков.

3) **Operator report**
   - в отчёте обработки очереди есть ссылка «Открыть страницу фестиваля»;
   - в отчёте есть ссылка «Открыть страницу Фестивали».

4) **Тестовая среда / разные Telegraph токены**
   - если `telegraph_path`/`fest_index_path` пришли из snapshot и создан(ы) под другим `TELEGRAPH_TOKEN`,
     то при `PAGE_ACCESS_DENIED` бот должен **создать новую** страницу под текущим токеном и сохранить новый `path/url`
     (не “падать” и не оставлять битые ссылки).

5) **Прогресс в UI (не молчать)**
   - при ручном запуске `/fest_queue` бот должен показывать прогресс (как минимум: сколько элементов найдено, какой элемент сейчас обрабатывается, и финальный итог);
   - для TG‑элементов очереди прогресс Kaggle/Telegram Monitoring должен быть виден оператору (а не только “тихий” финальный результат).

## Программа фестиваля: события vs program‑only

Создаём отдельное событие, если есть:
- дата, время и локация;
- хотя бы один «сильный» признак: уникальное название, отдельная площадка, отдельная регистрация/билет, явный тип (концерт/лекция/показ/спектакль) или длительность ≥ 45 минут.

В `festival.activities_json` остаются «program‑only» пункты:
- микропрограмма без времени/локации;
- развлечения без самостоятельного смысла («батуты/шарики/сахарная вата»);
- слабосигнальные пункты без уникального события.

## Иллюстрации и 3D‑превью

Правило обложки фестиваля:
- приоритет — `festival.preview_3d_url` (если появится);
- иначе `festival.photo_url`;
- иначе первая из `festival.photo_urls`.

Примечание про TG‑источники:
- если у фестиваля нет картинок в программе/событиях, но в исходном Telegram‑посте есть фото, очередь может (best‑effort) подхватить изображение из поста и сохранить его в `festival.photo_url/photo_urls` для выполнения DoD «обложка/галерея».
- если Catbox недоступен, загрузка картинок может фолбэкаться в Supabase Storage (нужны `SUPABASE_URL` + `SUPABASE_KEY`, bucket — `SUPABASE_MEDIA_BUCKET` или legacy `SUPABASE_BUCKET`).

Обложка должна отображаться:
- на странице фестиваля;
- на общей странице «Фестивали».

## Пролинковка между выпусками

Отдельной страницы серии нет. На странице выпуска должна быть навигация на другие выпуски этой серии (предыдущий/следующий/архив), отсортированные по датам.

## Источники серии (TG/VK)

Оператор может отметить источник как «фестивальный» и задать серию:
- в `/tg` для Telegram‑каналов;
- в списке VK‑сообществ.

Это **не означает**, что каждый пост фестивальный — это лишь сильный сигнал для матч‑логики.

## Operations

- Ручной запуск фестивальной очереди: `/fest_queue` (см. `docs/operations/commands.md`).
- Плановый запуск очереди описан в `docs/operations/cron.md`.
- Для аварийного fallback обложки индекса можно задать `FESTIVALS_INDEX_FALLBACK_COVER_URL` (должен быть preview‑friendly URL).
- Исторический fallback вида `https://telegra.ph/file/...` считается устаревшим (upload endpoint deprecated) и не используется как preview‑friendly цель.
